Контекст игры:
Это deck-building карточная игра. В игре есть личные колоды игроков, рука, сброс, уничтоженные карты, основная колода, барахолка из 5 открытых карт, стопка легенд, шальная магия, вялые палочки, фамильяры, планшеты колдунов, жетоны свойств и жетоны дохлых колдунов. Игроки начинают с 20 жизнями и стартовой колодой из 6 знаков, 1 палочки и 3 пшиков. В свой ход игрок разыгрывает карты, получает мощь, применяет эффекты, покупает карты, атакует врагов, может побеждать легенды. Игра заканчивается, если закончились легенды, не хватает карт для пополнения барахолки до 5 карт или закончилась стопка жетонов дохлых колдунов. Победа определяется по победным очкам в колоде с учётом штрафов.

ЭТАП 2. Рабочий прототип движка игры

Задача:
Создать headless-движок игры, который может симулировать партии без UI.

Что реализовать сначала:

1. Базовые модели:

   * Card;
   * Player;
   * GameState;
   * Market;
   * Deck / hand / discard / destroyed;
   * TurnContext;
   * Action;
   * GameConfig.
2. Поддержать игроков от 2 до 5.
3. Подготовка партии:

   * 20 жизней каждому игроку;
   * стартовая колода: 6 знаков, 1 палочка, 3 пшика;
   * раздача 5 карт;
   * основная колода;
   * барахолка 5 карт без начального беспредела;
   * стопка легенд по числу игроков;
   * стопки шальной магии и вялых палочек;
   * фамильяры как отдельный объект, который сначала лежит под планшетом и не действует, пока не куплен.
4. Игровой цикл:

   * пополнение барахолки до 5;
   * обработка беспредела при выходе на барахолку;
   * эффекты начала хода;
   * розыгрыш карт с руки в произвольном порядке;
   * накопление мощи;
   * покупка карт с барахолки, шальной магии, легенды и фамильяра;
   * помещение купленных/полученных карт в сброс, если не сказано иного;
   * конец хода;
   * сброс руки;
   * эффекты конца хода;
   * сброс сыгранных непостоянных карт;
   * добор 5 карт;
   * раскрытие новой легенды и групповая атака;
   * переход хода.
5. Перемешивание:

   * если нужно взять карту из пустой колоды, немедленно перемешать сброс и создать новую колоду.
6. Постоянки:

   * при розыгрыше остаются в игре;
   * действуют, пока находятся под контролем игрока;
   * не сбрасываются в конце хода;
   * могут быть сброшены/уничтожены эффектами.
7. Атака и защита:

   * атака может иметь выбранную цель, всех врагов, левого/правого врага, самого хилого/могучего и т.д.;
   * цель может применить защиту с руки;
   * защита не отменяет остальные свойства карты;
   * фамильяры-защиты могут перенаправлять эффект атаки на атакующего;
   * при групповом эффекте порядок применения — по часовой стрелке от указанного правилами игрока.
8. Легенды:

   * можно победить только одну легенду за ход;
   * новая легенда раскрывается в конце хода;
   * при раскрытии применяется групповая атака;
   * карта побеждённой легенды попадает в сброс и позднее играется как обычная карта с верхним эффектом, без групповой атаки.
9. Условия конца игры:

   * закончилась стопка легенд;
   * основная колода не может пополнить барахолку до 5;
   * закончились жетоны дохлых колдунов.
10. Подсчёт очков:

* победные очки на картах;
* штрафы от дохлых колдунов;
* штрафы от вялых палочек;
* особые VP-эффекты карт как расширяемые функции;
* tie-breaker: больше легенд, затем меньше жетонов дохлых колдунов.

Важный архитектурный принцип:
Эффекты карт должны быть реализованы через реестр эффектов, а не через огромную цепочку if/else. Например:

```
effect_registry = {
  "gain_power_1": GainPower(1),
  "draw_1": DrawCards(1),
  "attack_damage_6_left_or_right": AttackDamage(...),
}
```

Для карт с ещё не реализованным текстом используй fallback:

* карта даёт только явно распознанную мощь;
* сложный текст помечается как `not_implemented`;
* при симуляции можно включать режим `strict=True`, где такие карты вызывают ошибку, или `strict=False`, где они логируются и пропускаются.

Критерии готовности этапа 2:

* Есть команда `python -m src.game.simulate --players 3 --games 10`.
* Партия может завершиться без падения.
* Движок сериализует состояние в JSON.
* Есть RandomAgent, который выбирает легальные действия.
* Есть тесты на setup, ход, покупку, добор, барахолку, конец игры, подсчёт очков.
* Есть `docs/engine_architecture.md`.

ЭТАП 3. Агенты и обучение

Задача:
Добавить интерфейс агентов и подготовить среду для обучения.

Что реализовать:

1. Общий интерфейс агента:

```
class Agent:
    def choose_action(self, observation, legal_actions) -> Action:
        ...
```

2. Типы агентов:

   * RandomAgent;
   * GreedyBuyAgent: покупает самые дорогие/ценные карты;
   * AggressiveAgent: приоритет атак и добивания врагов;
   * EconomyAgent: приоритет мощи и добора;
   * HeuristicAgent: смешанная стратегия;
   * HumanCliAgent для ручной игры в консоли.
3. Observation space:
   Сформируй наблюдение так, чтобы агент не видел скрытую информацию:

   * своя рука;
   * свой сброс;
   * свои постоянки;
   * свои жизни;
   * доступная барахолка;
   * открытая легенда;
   * публичные зоны других игроков;
   * размеры колод/рук/сбросов врагов;
   * открытые эффекты.
4. Action space:

   * сыграть карту из руки;
   * купить карту;
   * выбрать цель атаки;
   * выбрать защиту;
   * выбрать карту для сброса/уничтожения;
   * закончить фазу;
   * закончить ход.
5. Создай Gymnasium/PettingZoo-compatible среду:

   * reset;
   * step;
   * observe;
   * legal action mask;
   * reward;
   * done/truncated;
   * info.
6. Reward design:
   Сделай минимум 3 режима:

   * `win_loss`: +1 за победу, 0 или -1 за поражение;
   * `vp_delta`: промежуточная награда за рост VP-оценки;
   * `hybrid`: победа + промежуточные сигналы за покупку полезных карт, урон, добор, удаление мусора.
7. Self-play:

   * скрипт обучения;
   * сохранение моделей;
   * оценка против RandomAgent и HeuristicAgent;
   * метрики winrate, средний VP, средняя длина партии, частота падений/ошибок.
8. Логирование:

   * сохранять replay партии;
   * сохранять seed;
   * уметь воспроизвести баг по seed.
9. Добавь benchmark:

   * `python -m src.agents.evaluate --agent heuristic --opponents random --games 100`.

Критерии готовности этапа 3:

* Есть несколько scripted-агентов.
* Есть RL environment с legal action mask.
* Есть self-play training skeleton.
* Есть evaluation script.
* Можно запустить серию партий и получить таблицу результатов.
* Документированы ограничения текущей модели.
