# Решения

## 2026-06-21. Минимальный стек импорта

- Используем Python 3.11+ как основной язык проекта.
- Для рендера PDF выбран PyMuPDF: он работает напрямую с локальными PDF и не требует системного Poppler.
- Для предобработки изображений выбран Pillow: достаточно для первого воспроизводимого пайплайна.
- OCR сделан опциональным. Без Tesseract карты не распознаются по догадке, а попадают в `needs_review`.
- Сегментация сначала реализована как projection/grid алгоритм без OpenCV, чтобы снизить входной порог запуска.

## 2026-06-21. Источник карт для движка

- Для этапа 2 используем `data/processed/cards_full.xlsx`, подготовленный вручную пользователем.
- Excel читается через стандартный OOXML ZIP/XML-парсер, чтобы не добавлять обязательную зависимость `openpyxl`.
- Нормализованная копия сохраняется в `data/processed/cards_full.json`.
- Эффекты карт не реализуются через цепочку `if/else`: используется реестр эффектов в `src.game.effects`.
- Первая версия применяет только безопасно извлекаемые базовые эффекты из текста: мощь, простой добор и простой урон.
- Сложные эффекты логируются/пропускаются в `strict=False` и выбрасывают ошибку в `strict=True`.

## 2026-06-22. Фазовый движок этапа 2.5

- В `GameState` добавлена явная `phase`, `pending_choice`, `effect_queue` и `dead_wizard_stack`.
- Legal actions вынесены в `src.game.legal_actions`, чтобы агенты не догадывались о допустимых действиях.
- Эффекты разложены на primitive effects; сложные тексты остаются видимыми как `partial`/`not_implemented`.
- Defense window моделируется как отдельная фаза, а не как скрытая проверка внутри урона.
- Смерть временно восстанавливает игрока до `GameConfig.death_reset_health = 20`; точное правило помечено как `TODO_RULE_CLARIFICATION`.
- Промежуточные PNG импорта исключены из git через `.gitignore`, потому что воспроизводятся командой импорта и занимают много места.

## 2026-06-22. Deck Manifest и CardInstance этапа 2.6

- Тиражи вынесены в `data/processed/deck_manifest.json`, чтобы отделить данные уникальных карт от физического состава партии.
- Стартовые карты, шальная магия, вялые палочки, легенды и дохлые колдуны используют явные правила stage 2.6.
- Основная колода в manifest содержит по 1 копии каждой уникальной основной карты и беспредела: официального тиража в `cards_full.xlsx` пока нет.
- Введен `CardInstance`: зоны игры хранят физические `instance_id`, а `CardDefinition` остается описанием уникальной карты.
- Legal actions теперь содержат `instance_id` для однозначного выбора конкретной карты.
- Текст карты разбирается через `src.game.card_text`; main/attack/defense/ongoing/group/scoring секции применяются раздельно.
- Coverage использует статусы `implemented`, `implemented_with_tests`, `partial`, `not_implemented`, `no_effect`, `data_error`.
- Replay на этом этапе только записывается; playback оставлен для следующих этапов.

## 2026-06-22. Проверочный слой этапа 2.6.1

- Основная колода в `data/processed/deck_manifest.json` валидируется как 124 физические карты: 26 беспределов по одной копии, 18 одиночных не-беспределов и 40 уникальных карт по две копии.
- `GameConfig` больше не содержит тиражи служебных стопок; шальная магия, вялые палочки и дохлые колдуны берутся из manifest, чтобы не было двух источников истины.
- `strict=True` использует тот же разбор простых эффектов, что и coverage: стартовые карты и полностью покрытые базовые шаблоны не должны падать как нереализованные.
- Групповая атака легенды применяется ко всем колдунам, включая текущего игрока; порядок целей начинается со следующего игрока по кругу.
- Replay остается write-only, но для диагностики добавлена команда `python -m src.game.replay_summary <replay.json>`.
